### 1. Project Title
**MeetMinds: AI-Powered Meeting Minutes & Knowledge Hub**

### 2. Real-World Context / Business Background
In the modern remote-work era, teams spend hours on Zoom, Teams, or Google Meet. While these platforms generate transcripts, nobody has the time to read through 5 pages of text to find out "What was decided about the database schema?" or "Who was assigned the frontend task?". Important decisions get lost in the noise, and actionable tasks fall through the cracks because manual note-taking is prone to human error.

### 3. Problem Statement
Build **MeetMinds**, a meeting intelligence dashboard. The application allows users to upload raw text transcripts from meetings. The system must use **LLMs** to automatically clean the text, generate structured **Minutes of the Meeting (MoM)**, and extract a list of **Action Items** with assignees. The core innovation is a **RAG-based "Meeting Brain"** that enables users to ask questions across *all* past meetings (e.g., *"When did we last discuss the API budget?"*), retrieving the exact context from weeks ago.

### 4. Target Users
- **Project Managers:** Tracking deliverables and decisions.
- **Development Teams:** Recalling technical decisions made in standups.
- **Student Capstone Teams:** Managing their own project discussions and tasks.

### 5. Core AI Features
The project must include the following 5 AI-driven features:

1.  **Automated Minutes of Meeting (MoM) Generator**
    *   Input raw transcript text (which is often messy and contains filler words like "um," "ah").
    *   The AI summarizes the discussion into clear sections: **Key Discussion Points**, **Decisions Made**, and **Open Questions**.
2.  **Action Item & Owner Extraction**
    *   Analyze the text to identify commitments.
    *   Extract structured data: `{ "Task": "Fix login bug", "Owner": "Rahul", "Deadline": "Friday" }`.
    *   Display these as a checklist on the dashboard.
3.  **Semantic "Memory" Search (The Meeting Brain)**
    *   Users can ask: *"What did we decide about the color scheme last month?"*
    *   The system uses **Pinecone** to search across *multiple* past meeting transcripts to find the relevant discussion, even if the user doesn't remember which meeting it was in.
4.  **"Unresolved Topic" Detector**
    *   The AI compares the current meeting with previous ones. If a topic (e.g., "AWS Migration") was raised in Meeting A but not marked as "Resolved" in Meeting B or C, the AI flags it: *"Warning: AWS Migration is still unresolved from 2 weeks ago."*
5.  **Follow-up Email Drafter**
    *   One-click generation of a professional email to all attendees, summarizing the meeting and listing their specific tasks, ready to be sent.

### 6. High-Level System Architecture

*   **Frontend (React):** A dashboard with a "Meeting History" list, a detailed "Transcript vs. Summary" view, and a Global Search bar.
*   **Backend (Node.js + Express):** APIs to handle text processing, vectorization, and saving meeting data.
*   **Database (MongoDB Atlas):**
    *   Stores: Structured Meeting Metadata (Date, Attendees, Title), the Generated MoM, and Action Items.
*   **Vector Database (Pinecone):**
    *   Stores: Embeddings of **Transcript Segments**. This allows the "Time Travel" search capabilities.
*   **AI Engine (Google AI Studio - Gemini):**
    *   Used for: Summarization, Entity Extraction (People/Tasks), and Cross-meeting RAG.

### 7. AI & RAG Workflow Explanation

**Phase A: Ingestion (After the meeting)**
1.  **Input:** User pastes the text transcript from Google Meet/Zoom.
2.  **Processing:** Node.js cleans the text (removing timestamps/speaker labels if needed).
3.  **Generation:** Gemini analyzes the text to create the MoM and Action Items JSON.
4.  **Embedding:** The full transcript is split into chunks (e.g., by topic change). Gemini creates vectors.
5.  **Storage:**
    *   *MongoDB:* Saves the Summary and JSON Tasks.
    *   *Pinecone:* Saves the vectors with metadata `{ meetingId: "101", date: "2023-10-05" }`.

**Phase B: Knowledge Retrieval (The RAG Flow)**
1.  **User Query:** *"Did Sarah agree to the timeline?"*
2.  **Retrieval:** The system embeds the query and searches Pinecone. It finds a chunk from "Meeting #4" and a chunk from "Meeting #6".
3.  **Augmentation:**
    *   *Context:* (Text from Meeting #4: "Sarah said she needs 2 weeks") + (Text from Meeting #6: "Sarah confirmed timeline is fine").
    *   *Question:* "Did Sarah agree?"
4.  **Generation:** Gemini synthesizes the answer: *"Yes, initially she asked for 2 weeks in Meeting #4, but confirmed the timeline was acceptable in Meeting #6."*

### 8. Expected Learning Outcomes
*   **Text Cleaning & Summarization:** Handling messy spoken language (transcripts) vs. clean written language.
*   **Temporal RAG:** Understanding how to search data that changes over time (what we said *yesterday* vs *today*).
*   **Entity Extraction:** Identifying names (Assignees) and dates within text.
*   **Information Synthesis:** Combining insights from multiple documents (meetings) into one answer.

### 9. Constraints & Assumptions
*   **Input Data:** Do **not** implement real-time Audio-to-Text (too complex for the workshop). Assume the user copies the transcript from Zoom/Teams/Otter.ai and pastes it into the app.
*   **Speaker Diarization:** Assume the input text already has "Speaker Name:" labels (standard in most transcript tools) so the AI knows who said what.
*   **Privacy:** Since this is a student project, use dummy meeting logs (e.g., "Project Kickoff", "Weekly Standup").
