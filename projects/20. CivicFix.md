### 1. Project Title
**CivicFix: AI-Powered Citizen Grievance & Resolution Platform**

### 2. Real-World Context / Business Background
Citizens often struggle to report local issues (e.g., potholes, uncollected garbage, broken streetlights) because government portals are confusing. Users don't know which department handles "drainage" versus "sewage," leading to misrouted complaints. On the other side, municipal officers are overwhelmed by duplicate reports for the same issue (e.g., 50 people reporting the same broken light). Cities need an intelligent layer that automates the categorization, routing, and prioritization of public grievances.

### 3. Problem Statement
Develop **CivicFix**, a smart city reporting dashboard. Citizens can report issues by uploading a photo and a brief text description. The system must use **Multimodal AI** to automatically identify the issue type from the image (e.g., "Pothole" vs "Garbage Dump"). Using **Semantic Search**, it routes the complaint to the correct Department (e.g., Public Works vs. Health) and detects **Duplicate Complaints** to prevent clutter. Finally, it uses **RAG** to fetch the specific **SLA (Service Level Agreement)** from the City Charter to tell the citizen exactly when to expect a resolution.

### 4. Target Users
- **Citizens:** Who want an easy way to fix their neighborhood.
- **Municipal Officers:** Who need a clean, de-duplicated list of tasks.
- **City Admins:** Tracking department performance.

### 5. Core AI Features
The project must include the following 5 AI-driven features:

1.  **"Snap & Categorize" (Multimodal Classification)**
    *   *Input:* User uploads a photo of a broken street sign.
    *   *Tech:* Use **Gemini Vision** to analyze the image and text description.
    *   *Output:* Auto-tag the issue (Category: "Infrastructure", Severity: "Medium") without the user manually selecting complex dropdowns.
2.  **Smart Department Routing (Semantic Search)**
    *   *Problem:* Users type "Water is logging on the street." (Is this Road Dept or Water Dept?)
    *   *Tech:* The system embeds the complaint and queries **Pinecone** against a "Department Responsibility" index to route it to the correct department (e.g., "Stormwater Drainage Dept") automatically.
3.  **Duplicate Incident Detector**
    *   *Function:* Before creating a new ticket, the AI checks if a *similar* complaint (semantically similar description + visually similar image context) was reported in the same location recently.
    *   *Output:* "This issue has already been reported by 3 neighbors. We have upvoted the existing ticket for you."
4.  **SLA & Rights Estimator (RAG)**
    *   *Input:* A "Pothole" complaint.
    *   *Tech:* The system retrieves the "City Citizen Charter" (PDF) from the vector DB.
    *   *Output:* "According to the Municipal Act Section 4.b, potholes must be filled within 72 hours. Your expected resolution date is [Date]."
5.  **Priority "Heatmap" Generator**
    *   The AI analyzes the sentiment and urgency of all incoming text descriptions.
    *   It identifies "High Priority Zones" (e.g., "10 reports of open live wires in Sector 4") and pushes a "Critical Alert" to the Admin Dashboard.

### 6. High-Level System Architecture

*   **Frontend (React):** A mobile-friendly Reporting View (Camera + Location), and an Admin Dashboard (Map View + Kanban Board).
*   **Backend (Node.js + Express):** APIs for image handling, vector search, and ticket management.
*   **Database (MongoDB Atlas):**
    *   Stores: Incident Reports (Location, Status, Image URL), Department Info, and User Profiles.
*   **Vector Database (Pinecone):**
    *   Stores: Embeddings of **Department Descriptions** (for routing), **City Charter/Rules** (for SLAs), and **Past Incident Descriptions** (for duplicate detection).
*   **AI Engine (Google AI Studio - Gemini):**
    *   Used for: Image recognition, Routing logic, RAG-based SLA retrieval, and Severity scoring.

### 7. AI & RAG Workflow Explanation

**Phase A: Knowledge Base Ingestion**
1.  **Source:** Admin uploads "City_Charter.pdf" (Rules) and "Dept_Roles.txt" (Routing info).
2.  **Chunking & Embedding:** Split text and create vectors.
3.  **Storage:** Pinecone stores vectors.

**Phase B: The Reporting Flow (Multimodal + RAG)**
1.  **User Action:** Uploads photo of overflowing garbage. Writes: "Smells bad, been here 3 days."
2.  **Visual Analysis:** Gemini Vision identifies: "Garbage Pile / Waste Management."
3.  **Duplicate Check:** System queries Pinecone for "garbage smell" near this location. (Result: No match found).
4.  **Routing:** System matches "Garbage" to "Sanitation Department."
5.  **SLA Retrieval (RAG):**
    *   *Query:* "Garbage collection time limit."
    *   *Context:* (Retrieved from Charter: "Solid waste must be cleared within 24 hours of reporting").
6.  **Ticket Creation:** System saves the ticket to MongoDB assigned to "Sanitation."
7.  **User Response:** "Ticket #901 created. Assigned to Sanitation. Expected resolution: 24 Hours (per City Charter)."

### 8. Expected Learning Outcomes
*   **Multimodal Pipelines:** processing Image + Text + Location data together.
*   **Classification vs Clustering:** Using vectors to route tickets (Classification) and find duplicates (Clustering/Similarity).
*   **Geospatial Concepts:** Handling location data alongside AI (e.g., "Find duplicates *near* this lat/long").
*   **Public Sector Logic:** Building systems that reference "Rules/Laws" (SLAs) to hold authorities accountable.

### 9. Constraints & Assumptions
*   **Location:** Use hardcoded lat/long or browser Geolocation API. For duplicate detection, strict location matching isn't needed; just check if the text description is semantically similar to recent tickets.
*   **Images:** Use sample images of potholes/garbage for testing.
*   **Data:** Create a dummy "City Charter" text file with simple rules (Pothole=3 days, Light=2 days).

### 10. Optional Extensions (For Advanced Students)
*   **Auto-Email to Official:** When a ticket is created, use `nodemailer` to send a formatted email to the "Department Head" with the AI-generated summary.
*   **"Fixed" Verification:** When the official uploads a "Resolved" photo, use Gemini Vision to compare the "Before" and "After" photos to verify if the work was actually done.
*   **Twitter Bot:** Automatically tweet the complaint to the city handle if the SLA time is breached.
